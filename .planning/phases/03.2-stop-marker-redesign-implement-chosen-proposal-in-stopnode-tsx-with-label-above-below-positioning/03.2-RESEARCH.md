## Summary

Phase 03.2 replaces the existing `MarkerPin` SVG map pin in `StopNode.tsx` with a Build Progress Bricks SVG marker. The key technical scope is: (1) an inline SVG brick marker whose geometry is derived from the existing `icon.svg` favicon, (2) five CSS-driven visual states, (3) a brick-place keyframe animation for the active entrance, (4) a color-sweep CSS transition from active to visited, and (5) a `labelPosition` field added to the `Stop` type and data files to control above/below label placement.

The work is self-contained within `StopNode.tsx` plus type and data changes. The `MapCanvas.tsx` parent already passes `isActive` and `isVisited` props correctly — no parent changes are needed. The only external files that change are `src/types/presentation.ts` (add `labelPosition`) and all five stop data files (add the value).

The mockup in `.planning/assets/mockups/stop-marker-states.html` is a fully-worked JavaScript SVG generator that already solves all the hard geometry questions. The planner can treat the mockup's coordinate system (`VB_W=48, VB_H=66`, the `POS` and `STUD` constants, and the `createMarkerSVG` function logic) as the definitive SVG blueprint — the task is to translate it into static React/TypeScript, not re-invent the geometry.

**Primary recommendation:** Implement the brick marker as a pure SVG function component that accepts `stopIndex` and `state` props. All state-driven transforms go on the `<button>` wrapper via CSS classes. Animate active entrance with a CSS `@keyframes` rule. Animate the visited color sweep with CSS `transition: fill` on the SVG elements. No Motion library involvement — this is pure CSS territory.

### Pattern 3: CSS @keyframes for Brick-Place Animation

**What:** When a stop becomes active, the newest (topmost) brick animates in from above. Use a CSS class added to the topmost brick element, playing a `@keyframes` animation on mount.

**Example:**
```css
/* In globals.css or a <style> block in StopNode.tsx via CSS modules */
@keyframes brick-drop {
  from {
    opacity: 0;
    transform: translateY(-8px);
  }
  to {
    opacity: 1;
    transform: translateY(0);
  }
}

.brick-drop-in {
  animation: brick-drop 200ms cubic-bezier(0.34, 1.56, 0.64, 1) forwards;
}

@media (prefers-reduced-motion: reduce) {
  .brick-drop-in {
    animation: none;
  }
}
```

**Implementation note:** The animation only fires when entering the active state. Since React re-renders the SVG with a new key or class when `isActive` changes to true, the animation will trigger automatically. Using `animation-fill-mode: forwards` ensures the brick stays in final position.

### Pattern 4: Color Sweep via CSS transition on SVG fill

**What:** When transitioning from active to visited, all brick fills change from the active palette to uniform green. CSS `transition: fill 300ms ease` on each `<rect>` element will produce the sweep effect.

**Example:**
```typescript
// Apply to every rect in the SVG:
<rect
  x={...} y={...} width={...} height={...}
  fill={brickFill}
  style={{ transition: 'fill 300ms ease' }}
/>
```

**Trade-off:** When using React with inline styles, `fill` transitions work because SVG presentation attributes that are also CSS properties (`fill`, `stroke`, `opacity`) respond to CSS transitions. This is confirmed by CSS Transitions spec — `fill` is an animatable property for SVG elements.

### Pattern 5: State Variant Computation

**What:** Derive the visual state from `isActive`, `isVisited` props plus hover/focus event handlers.

```typescript
// State priority: active > hover/focus > visited > default
// Hover and focus are tracked with local useState
const [isHovered, setIsHovered] = useState(false)
const [isFocused, setIsFocused] = useState(false)

const variant: BrickState =
  isActive    ? 'active'   :
  isFocused   ? 'focus'    :
  isHovered   ? 'hover'    :
  isVisited   ? 'visited'  :
                'default'
```

### Anti-Patterns to Avoid

- **Putting hover/focus state in the parent `MapCanvas`:** Keep it local to `StopNode`. The parent should never need to know about hover state.
- **Using the ui `Button` component:** Locked decision — raw `<button>` element stays. The brick marker is a custom map marker, not a standard action button. The Button component's border-bottom depth effect and fixed dimensions would break the SVG layout.
- **CSS class-based fill control via Tailwind:** SVG fill colors cannot be represented as Tailwind v4 tokens for the custom brick colors. Use inline `fill` attributes on SVG elements directly.
- **`overflow: hidden` on the button wrapper:** The marker uses `overflow: visible` on the SVG so drop-shadows and the scale transform can extend beyond the bounding box. Never clip the container.
- **Using `transform: translate(-50%, -50%)` if scale from center is wanted:** The scale transform in hover/focus state will scale from the element's transform origin. Combine `transform-origin: center bottom` on the SVG wrapper so the brick stack grows upward and stays grounded at its base, not jumping.

## Common Pitfalls

### Pitfall 1: SVG `overflow: visible` not set, transforms clip

**What goes wrong:** The scale(1.1) hover transform and the drop-shadow filter extend beyond the SVG's bounding box. If the SVG or its container has `overflow: hidden` (the browser default for `<svg>` elements in some contexts), the effect gets clipped visually.

**Why it happens:** The mockup sets `overflow: visible` on SVG elements explicitly. React SVG elements default to `overflow: hidden`.

**How to avoid:** Add `overflow="visible"` to the `<svg>` element and ensure no ancestor has `overflow: hidden`.

**Warning signs:** Drop-shadow cuts off at the SVG edge; scale effect looks truncated on top.

---

### Pitfall 2: Brick-place animation fires on every re-render, not just state entry

**What goes wrong:** If the `BrickMarker` component re-renders for any reason while active, the brick-drop animation replays.

**Why it happens:** CSS animations replay whenever the element is remounted or the animation class is re-applied.

**How to avoid:** Apply the animation class only to the topmost brick (which is new when stop count increases). Since the brick layout is determined by `stopIndex` (static, never changes per stop), the "newest brick" is always a fixed element in the SVG. The animation class should be applied based on `isActive` state only — it will replay if the user re-clicks the same active stop, which is acceptable.

**Warning signs:** All bricks animate on every click even when stop hasn't changed.

---

### Pitfall 3: `transform` on `<button>` conflicts with `position: absolute`

**What goes wrong:** The button is absolutely positioned. Adding CSS transforms (scale, translateY) via className changes can cause layout issues if the transform-origin is not explicitly set.

**Why it happens:** By default, `transform-origin` is `50% 50%` (center). For a brick stack that should grow upward from its base, the transform origin should be `center bottom` so scale doesn't shift the anchor point of the marker on the map.

**How to avoid:** Set `transform-origin: center bottom` on the element receiving the scale transform (the inner wrapper `div`, not the `<button>` itself — the button has the `position: absolute` anchor).

**Warning signs:** Active or hovered markers visually drift from their map coordinate.

---

### Pitfall 4: `labelPosition` data field causes TypeScript errors on existing Stop instances

**What goes wrong:** Adding a required `labelPosition` field to the `Stop` type breaks all five existing stop data files until they're updated.

**Why it happens:** TypeScript strict mode, used in this project, enforces all interface fields.

**How to avoid:** Update `presentation.ts` and all five stop data files in the same plan/task. Do not make `labelPosition` optional (`labelPosition?: ...`) — it should be required to ensure no stop is accidentally omitted.

**Warning signs:** `tsc` type errors on all stop data files immediately after the type change.

---

### Pitfall 5: CSS `transition: fill` on SVG rects requires presentation attribute, not CSS class

**What goes wrong:** If `fill` is set via a CSS class (e.g., `className="fill-lego-red"`), CSS transitions will work. But if `fill` is set via the SVG attribute (`fill="#E3000B"`), transitioning from a CSS rule may conflict because attribute presentation values have lower CSS specificity than class-based values.

**Why it happens:** CSS specificity rules apply to SVG presentation attributes vs stylesheet declarations.

**How to avoid:** Use the inline `style={{ fill: '#E3000B', transition: 'fill 300ms ease' }}` pattern (not the SVG attribute `fill="#..."`) so transitions reliably work via CSSOM. OR use the SVG attribute and accept that transitions may not fire in all browsers.

**Best practice:** Use `style={{ fill: ..., transition: 'fill 300ms ease' }}` on all SVG `<rect>` elements.

---

### Pitfall 6: `data-state` attribute on button — existing tests check for it

**What goes wrong:** The existing `StopNode.test.tsx` checks `data-state` on the button element. If the rewritten component removes or renames this attribute, 4 existing tests that rely on it for DOM state assertions may break.

**Why it happens:** The current `StopNode.tsx` sets `data-state={variant}` on the button. Tests reference `data-state` implicitly through the button's rendered class differences.

**How to avoid:** Keep `data-state` on the button element in the rewritten component. The existing test checks `activeBtn?.className !== visitedBtn?.className` — the data attribute itself is not directly asserted, but keeping `data-state` is low-cost and maintains DOM debuggability.

**Warning signs:** Existing 8 StopNode tests fail after rewrite.

```typescript
// StopNode.tsx (new BrickMarker function component)
type BrickState = 'default' | 'hover' | 'focus' | 'active' | 'visited'

function BrickMarker({ stopIndex, state }: { stopIndex: number; state: BrickState }) {
  const layout = BRICK_LAYOUTS[stopIndex]
  const topLevel = layout.level3 ? 3 : layout.level2 ? 2 : 1

  function getColor(level: 1 | 2 | 3, side?: 'left' | 'right') {
    if (state === 'active') {
      if (level === 1) return COLORS.red
      if (level === 2 && side === 'left')  return COLORS.yellow
      if (level === 2 && side === 'right') return COLORS.blue
      if (level === 3 && side === 'left')  return COLORS.red
      if (level === 3 && side === 'right') return COLORS.green
    }
    if (state === 'visited') return COLORS.green
    return COLORS.grey
  }

  const plateColor = (state === 'active' || state === 'visited') ? PLATE.green : PLATE.grey

  return (
    <svg
      width={VB_W} height={VB_H}
      viewBox={`0 0 ${VB_W} ${VB_H}`}
      fill="none"
      overflow="visible"
      aria-hidden="true"
    >
      {/* Baseplate */}
      {/* Level 1 full brick */}
      {/* Level 2/3 half bricks */}
      {/* Studs on topmost level */}
    </svg>
  )
}
```

## Open Questions

1. **Transform anchor for absolutley-positioned button**
   - What we know: current code uses `translate(-50%, -100%)` anchoring the marker tip at the stop coordinate
   - What's unclear: with the new brick marker (no pointed tip), whether to anchor by the baseplate bottom (`-100%`) or center (`-50%`)
   - Recommendation: anchor by baseplate bottom (`translate(-50%, -100%)`) — most natural for a stack sitting on the map point. The label-above stops will render the label further above the map pin location, which is visually clear.

2. **Where to define brick-drop @keyframes**
   - What we know: globals.css is the project's CSS home; but keyframes scoped to StopNode could live in a CSS Module
   - What's unclear: project currently has no CSS Modules setup
   - Recommendation: Add to `globals.css` `@layer components` — consistent with existing `.stud-pattern` pattern there. CSS Module setup is not present and not needed.

3. **Testing the color sweep transition**
   - What we know: jsdom cannot test CSS transitions (confirmed by existing Phase 03.1 learnings)
   - What's unclear: whether the test suite should assert `style={{ transition: 'fill ...' }}` as a DOM attribute
   - Recommendation: Add a structural test that asserts SVG rects have the `transition` style when `isVisited=true`. This is testable in jsdom via `element.style.transition`.
