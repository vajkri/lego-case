## Summary

Phase 03.3 redesigns the footer progress wizard in `PresentationFooter.tsx`. The current implementation (lines 32–61) uses generic grey/amber/green rounded circles with a tick or number label and a 2px connector line. The replacement uses inline SVG LEGO minifig heads with three distinct facial expressions driven purely by presentation state, thicker connectors with semantic color coding, sub-slide progress dots under the current stop, and removes the redundant `MapProgressIndicator` corner widget.

The technical work is straightforward React component work: no new external dependencies are needed. The patterns are directly inherited from Phase 03.2 (BrickMarker, StopNode) — CSS `transition` on SVG `fill` via inline `style` prop, `onMouseEnter`/`onMouseLeave`/`onFocus`/`onBlur` for hover/focus state, and `data-state` attributes. The most novel element is the connector left-to-right sweep, which requires deciding between a `background` property transition approach and a `linear-gradient` masking approach.

The component promotion of `SubSlideProgress` from `SlideFrame/SubSlideProgress.tsx` to `ui/SubSlideProgress.tsx` is a refactor that adds a `size` prop and updates barrel exports. Existing tests for `SubSlideProgress` live in `SlideFrame.test.tsx` and will need updating to import from the new location.

**Primary recommendation:** Build a `MinifigHead` sub-component (self-contained, receives `state: 'default' | 'current' | 'visited'`) that encapsulates all SVG anatomy and color data. The `PresentationFooter` rewrite maps `stops` array to `<StopStep>` (head + sub-dots) separated by connectors, computing connector color from the index relationship to `currentStop` and `visitedStops`.

### Pattern 1: MinifigHead SVG Component

**What:** A pure presentational React component that renders a 34×40 viewBox SVG with anatomy driven by `state` prop. Face feature paths are hardcoded per expression — no data arrays.

**When to use:** Whenever a stop head is rendered (14 times in the footer for a 14-stop presentation).

**Example (derived from progress-track-states.html mockup):**

```tsx
// Source: .planning/assets/mockups/progress-track-states.html
type HeadState = 'default' | 'current' | 'visited'

const HEAD_COLORS = {
  default: { head: '#D4D4D8', stud: '#B5B5B9', neck: '#B5B5B9' },
  current: { head: '#F5CD2D', stud: '#E8C020', neck: '#D4A800' },
  visited: { head: '#00A850', stud: '#008040', neck: '#008040' },
} as const

interface MinifigHeadProps {
  state: HeadState
  width?: number
  height?: number
}

function MinifigHead({ state, width = 28, height = 34 }: MinifigHeadProps) {
  const c = HEAD_COLORS[state]
  return (
    <svg width={width} height={height} viewBox="0 0 34 40" fill="none" aria-hidden="true">
      {/* Stud — flat cylinder, rx=1 */}
      <rect x="11" y="2" width="12" height="4" rx="1"
        style={{ fill: c.stud, transition: 'fill 200ms ease' }} />
      {/* Head body */}
      <rect x="3" y="6" width="28" height="26" rx="8"
        style={{ fill: c.head, transition: 'fill 200ms ease' }} />
      {/* Neck — wider than stud */}
      <rect x="9.8" y="32" width="14.4" height="4" rx="1"
        style={{ fill: c.neck, transition: 'fill 200ms ease' }} />
      {/* Face features — conditionally rendered per state, no transition */}
      {state === 'default' && <DefaultFace />}
      {state === 'current' && <CurrentFace />}
      {state === 'visited' && <VisitedFace />}
    </svg>
  )
}
```

**Key: CSS `transition` on `fill` via inline `style` — matches Phase 03.2 BrickMarker pattern exactly.** Face feature elements have no transition (instant swap matches brief).

### Pattern 2: State-Computed Connector Color

**What:** Per-connector color computed from the connector's position relative to `currentStop` and `visitedStops`.

**When to use:** Each connector between stop i-1 and stop i.

```tsx
// Connector color logic (connector at index i sits between stop i-1 and stop i)
function getConnectorColor(
  connectorIndex: number,  // 1-based: connector between stop[i-1] and stop[i]
  currentStop: number,
  visitedStops: number[]
): string {
  // Connector leads INTO currentStop from the left
  if (connectorIndex === currentStop) return 'var(--color-lego-red)'
  // Connector is between two visited stops
  if (connectorIndex <= currentStop && visitedStops.includes(connectorIndex - 1)) {
    return 'var(--color-lego-green)'
  }
  // Default: upcoming
  return 'var(--color-lego-grey-pale)'
}
```

### Pattern 5: Connector Left-to-Right Sweep (Claude's Discretion)

Two viable approaches:

**Option A — `background` color transition (simpler):**
```css
.connector {
  transition: background-color 200ms ease;
}
```
This is an instant color swap, not directional. Simpler but doesn't achieve the "sweep" effect.

**Option B — `linear-gradient` + `background-size` sweep (achieves sweep):**
```css
.connector {
  background: linear-gradient(to right, var(--color-lego-green) 50%, var(--color-lego-grey-pale) 50%);
  background-size: 200% 100%;
  background-position: right;
  transition: background-position 200ms ease-out;
}
.connector.visited {
  background-position: left;
}
```
This achieves a true left-to-right fill. Slightly more complex but visually correct.

**Recommendation:** Use Option B for the connectors that transition to `visited`. For `current` (red) vs `default` (grey), a simple `background-color` transition suffices because red connectors don't sweep — they appear with the head state change.

### Anti-Patterns to Avoid

- **Animating SVG `fill` attribute (not `style.fill`):** React will not apply CSS transitions to SVG `fill` attributes; must use inline `style={{ fill: value }}` with the transition on that style property. Phase 03.2 confirmed this.
- **Using Motion library for color fades:** Overkill; CSS transition is sufficient and matches existing pattern.
- **Rendering sub-dots under all stops:** Only render under the current stop. An empty spacer div of fixed height should sit under other stops so the track doesn't shift vertically.
- **Importing SubSlideProgress from SlideFrame after promotion:** After moving the file, `SlideFrame/SubSlideProgress.tsx` must become a re-export shim so existing imports and tests don't break until cleaned up.

## Common Pitfalls

### Pitfall 1: SVG `fill` transition requires inline `style` prop
**What goes wrong:** Developer writes `fill="#D4D4D8"` as SVG attribute. Color changes instantly, no transition.
**Why it happens:** CSS `transition` only applies to CSS properties. SVG `fill` as an attribute is not a CSS property.
**How to avoid:** Always use `style={{ fill: color, transition: 'fill 200ms ease' }}` on SVG rect elements. Phase 03.2 already established this; copy the pattern.
**Warning signs:** Colors change abruptly without fade on state switch.

### Pitfall 2: Sub-slide dot container height shift
**What goes wrong:** Footer height changes between stops because the sub-dots area appears/disappears under the current stop but not others.
**Why it happens:** The `stop-step` column height changes when sub-dots are shown vs hidden.
**How to avoid:** Reserve the sub-dots space with a fixed-height placeholder (`min-h-[8px]` or equivalent) under every stop. Only the current stop renders actual dots; others render an empty spacer of the same height.
**Warning signs:** Footer jumps vertically when navigating between stops.

### Pitfall 3: `--color-lego-green-dark` not available as Tailwind class before token is added
**What goes wrong:** Using `fill-lego-green-dark` Tailwind class before adding the token to `@theme`.
**Why it happens:** Token doesn't exist in globals.css yet.
**How to avoid:** Add `--color-lego-green-dark: #008040` to `@theme` in `globals.css` as the very first task. Use it as an inline `style` value (not Tailwind class) for SVG fill anyway — consistent with how all SVG fills are done.
**Warning signs:** TypeScript/build error or wrong color value on visited head stud/neck.

### Pitfall 4: SubSlideProgress import path breakage after promotion
**What goes wrong:** `SlideFrame.test.tsx` imports `SubSlideProgress` from `../SubSlideProgress`. After file move, import fails.
**Why it happens:** Physical file moved from `SlideFrame/SubSlideProgress.tsx` to `ui/SubSlideProgress.tsx`.
**How to avoid:** Leave a re-export shim at `SlideFrame/SubSlideProgress.tsx` pointing to the new location, OR update the test import. Update `SlideFrame/index.ts` to re-export from the new location.
**Warning signs:** Test run fails with module-not-found error.

### Pitfall 5: Missing `aria-current="step"` on current stop head button
**What goes wrong:** Accessibility regression — screen readers can't identify the current step.
**Why it happens:** Forgetting to carry over the `aria-current` attribute from the old implementation.
**How to avoid:** Add `aria-current={i === currentStop ? 'step' : undefined}` to each head button.
**Warning signs:** A11Y audit flags missing step indicator.

### Pitfall 6: Connector color logic edge case at stop 0
**What goes wrong:** Connector at index 0 between stop 0 and... nothing. Off-by-one error.
**Why it happens:** Connectors are rendered between stops, so there are `n-1` connectors for `n` stops. Connector `i` sits between stop `i-1` and stop `i`.
**How to avoid:** Render connectors only for `i > 0` (same pattern as current `PresentationFooter.tsx` line 36: `{i > 0 && <Connector />}`). The connector before stop `i` is "current" if `i === currentStop`, "visited" if `i < currentStop`, else "default".
