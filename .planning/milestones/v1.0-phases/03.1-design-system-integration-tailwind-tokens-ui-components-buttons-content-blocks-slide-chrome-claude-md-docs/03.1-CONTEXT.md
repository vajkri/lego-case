<decisions>
## Implementation Decisions

### Design tokens
- All tokens from `styleguide.md` §1 defined in Tailwind v4 `@theme` block in `globals.css`
- Replaces current placeholder `--color-brand` and `--font-display` with full LEGO token set
- CSS custom properties: colors (primary + neutrals + tints), borders, radii, shadows, depth shadows
- No `tailwind.config.js` — CSS-first approach per Phase 1 decision

### Font loading
- Use `next/font` for Baloo 2 and DM Sans (not CDN `<link>` tags)
- Baloo 2: weights 500, 600, 700, 800 (display font)
- DM Sans: weights 400, 500, 600, italic 400 (body font)

### Slide overlay theme
- Switch from current dark (slate-900) to light theme
- Grey-wash (#F0EEE8) outer background, white inner card, dark text (#2C2C2C)
- Follow styleguide definitions exactly
- Stud dot pattern on the header bar ONLY — not on slide content area

### Slide layout — Option A (Minimal Header)
- Red stud-pattern header bar: LEGO logo placeholder (left) + close button (right) only
- Stop badge, sub-slide progress dots, slide heading, and content all live inside the white content card
- Reference mockup: `.planning/assets/mockups/slide-layout-comparison.html` (Option A)
- Reference styleguide: `.planning/assets/mockups/styleguide.html` (Full Slide Layout section)

### SlideFrame wrapper component
- Full chrome wrapper: owns stud-pattern header bar, grey-wash background, white inner card container, and nav arrows at page edges
- Children are the slide content (badge, progress, heading, content blocks)
- Phase 4 authors just pass content in as children
- Transitions (AnimatePresence zoom in/out) stay in existing OverlayPresence component — separate concern from layout

### Navigation arrows
- LEGO brick-style buttons replacing current semi-transparent circles
- Grey back arrow (left), red forward arrow (right)
- 48×48px, rounded-lg (12px), 4px depth border, brick-press active state
- Positioned at page edges (absolute left/right within slide body)

### Button component
- Three variants: yellow (primary action), red (forward/next), grey (back/secondary)
- Icon size (48×48px) and label size (h-48px, px-22px)
- Hover, active press (translateY(2px) + border shrink to 2px), disabled states
- All rounded-lg (12px) with 4px depth border

### Content block components
- Build all 6 types: BulletList, TwoColumnCards, EntityCards, NumberedSteps, CalloutBox, DataTable
- Variant system via `variant` prop: default, red, yellow (controls tint bg, border, depth shadow, accent)
- Depth shadow system: hard box-shadow (0 6px 0 0) — not border-bottom — to preserve rounded corners

### Stud dot pattern
- Reusable implementation (Claude's discretion on whether CSS utility, custom property, or component)
- Must be easily applied to any element
- Used sparingly: header bars and hero areas only

### Component refactoring (in this phase)
- **SlideHeader** → replaced by SlideFrame's header bar (stud pattern + logo + close)
- **PresentationFooter** → refactored to use new Button component (grey/yellow/red variants, brick depth). Light touch on progress track — Phase 03.3 redesigns it.
- **SlideNavArrows** → refactored to LEGO brick arrow buttons (grey left, red right)
- **SlideContent** → SKIP — Phase 4 addresses with frontend-design creativity

### CLAUDE.md documentation
- Document design system conventions for future phases
- Token usage, component APIs, variant system, barrel exports

### Claude's Discretion
- Stud pattern implementation approach (CSS utility class, custom property, or component wrapper)
- Content block component API design (prop shapes for each of the 6 types)
- Exact `next/font` configuration details
- Token naming in `@theme` block (how to map custom properties to Tailwind utilities)
- Whether to create `src/components/layout/` primitives or keep everything in `ui/`

</decisions>

<specifics>
## Specific Ideas

- "Use definitions in styleguide" — the styleguide.md and styleguide.html are the authoritative spec. Don't invent new patterns.
- The stud pattern is NOT part of the slide itself, only the page headline/header bar. The Full Slide Layout section in styleguide.html is the reference for how slides should look.
- Slide-inspiration screenshots show the Figma Make version (multi-color stops, back arrow, dark theme) — these are SUPERSEDED by the styleguide decisions (single red accent, no back arrow, light theme).
- User prefers Option A from the comparison mockup: `.planning/assets/mockups/slide-layout-comparison.html`

</specifics>

## Existing Code Insights

### Reusable Assets

- `SlideHeader.tsx`: Will be replaced by SlideFrame's stud-pattern header bar. Close button logic reusable.

### Established Patterns
- `'use client'` boundary on all interactive/animation components
- Barrel exports in each component directory (`index.ts`)
- Component folders follow CLAUDE.md: `ui/` for primitives, `features/{page}/` for feature components
- Tailwind v4 CSS-first: `@theme` block in `globals.css`, no config file
- State-driven: reducer handles transitions, components are display-only

### Integration Points
- `globals.css`: Replace placeholder `@theme` block with full LEGO token set
- `src/app/layout.tsx`: Add `next/font` imports, apply font CSS variables to `<html>` or `<body>`
- `src/components/ui/`: New directory for Button, content blocks, slide chrome primitives
- `src/components/features/slide/SlideOverlay/`: SlideOverlay refactored to compose SlideFrame
- `src/components/features/presentation/PresentationFooter.tsx`: Refactored to use Button component
