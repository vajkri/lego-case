# Phase 03.2: Stop Marker Redesign - Context

**Gathered:** 2026-03-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the current map pin marker in `StopNode.tsx` with Build Progress Bricks (Proposal #2). Each stop's brick count matches its position (1–5), with 5 visual states (default, hover, focus, active, visited), smart label positioning (above/below per stop), and subtle state transition animations. Reuses the favicon (`src/app/icon.svg`) as the visual reference for brick proportions and stud style.

Does NOT include: road/path visual changes (03.3), progress track redesign (03.3), overall map aesthetic (03.4), or slide content (Phase 4).

</domain>

<decisions>
## Implementation Decisions

### Chosen proposal
- **Build Progress Bricks** (Proposal #2 from `stop-marker-proposals.html`)
- Visual reference mockup: `.planning/assets/mockups/stop-marker-states.html`
- Favicon (`src/app/icon.svg`) is the primary reference for brick proportions and stud style

### Brick layout per stop
- Fixed height: baseplate + 3 levels (max). Brick count varies 1–5 by mixing full and half-width bricks.
- Half-width bricks are flush (no gap between left and right halves on the same level).
- Bricks never change form once placed — if a brick starts as half-width, it stays half-width across all stops.

| Stop | Level 1 (bottom) | Level 2 (middle) | Level 3 (top) |
|------|-----------------|-----------------|---------------|
| 1 | 1 full brick | empty | empty |
| 2 | 1 full brick | 1 half (left) | empty |
| 3 | 1 full brick | 2 halves (left + right) | empty |
| 4 | 1 full brick | 2 halves (left + right) | 1 half (left) |
| 5 | 1 full brick | 2 halves (left + right) | 2 halves (left + right) |

### Stud style
- Favicon-proportioned rounded rectangles with **no border-radius** (sharp rectangles)
- Studs present on ALL states including default
- Studs only on topmost bricks
- Full-brick studs positioned to match half-brick stud spacing (visually consistent when transitioning between levels)

### Five visual states

**Default (unvisited):**
- Full opacity grey bricks (#D4D4D8), grey baseplate (#B5B5B9)
- No transforms or shadows

**Hover (mouse enter):**
- Same grey colors, no color change
- `scale(1.1)` + drop-shadow
- CSS `transition: all 0.25s ease`

**Focus (keyboard tab):**
- Same grey colors
- 3px solid blue (#0055BF) outline with 4px offset + `scale(1.08)`
- Blue-tinted label
- CSS `transition: all 0.25s ease`

**Active (current stop):**
- Favicon multi-color palette per brick:
  - Level 1 (full): red (#E3000B)
  - Level 2 left: yellow (#F5CD2D)
  - Level 2 right: blue (#0055BF)
  - Level 3 left: red (#E3000B)
  - Level 3 right: green (#00A850)
- Baseplate: green (#00A850)
- `translateY(-3px)` + drop-shadow
- Brick-place entrance animation: newest brick drops in from above (translateY + opacity keyframe)
- Yellow label with bold text

**Visited:**
- Uniform green bricks (#00A850), green baseplate (#00A850)
- No transforms or shadows (settled)
- Color sweep transition from active colors to green (~300ms fade)

### Label positioning
- Manual `labelPosition: 'above' | 'below'` field per stop in data
- Stop 1: above
- Stop 2: above
- Stop 3: below
- Stop 4: above
- Stop 5: above

### State transition animations
- **Active entrance:** Brick-place keyframe — newest brick animates in from above (translateY + opacity)
- **Active → Visited:** Color sweep — smooth color fade from active palette to green (~300ms)
- **Hover/Focus:** CSS `transition: all` only — no JS animation
- **Timing:** Quick and snappy — all transitions under 250ms (except color sweep)
- Presentation context: don't make the audience wait

### Accessibility
- Raw `<button>` element stays (not the ui Button component — this is a custom SVG map marker, not a standard action button)
- Focusable, keyboard-navigable, screen-reader announced (existing A11Y from Phase 3)
- WCAG-compliant focus indicator (blue outline ring)
- Labels always visible (MAP-02)

### Claude's Discretion
- Exact SVG dimensions and coordinate values for the brick marker
- Brick-place keyframe implementation details (duration, easing, translateY distance)
- Color sweep transition implementation (CSS transition vs Motion animate)
- Whether to add design tokens for new colors (brick green on baseplate) or use inline values
- How to structure the `labelPosition` data field (in stop data files or as a separate constant)

</decisions>

<specifics>
## Specific Ideas

- "Re-use as much as possible from the favicon for the stop nodes" — the favicon IS a Build Progress Brick. Match its proportions, stud style, shadow technique (darker shade rect offset 1px down).
- The brick-place animation reinforces the "building" metaphor — each stop you visit adds a brick to the construction.
- Colors for stops 4-5 were specifically chosen to avoid same-color stacking: level 3 uses red + green on top of yellow + blue (level 2).
- Full mockup reference with all states × all stops: `.planning/assets/mockups/stop-marker-states.html`

</specifics>
